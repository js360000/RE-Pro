Cross-platform reverse engineering, source recovery, and rebuild prep

Turn opaque binaries into readable evidence, recoverable source, and actionable rebuild workflows.

RE-Pro is a Python-based reverse-engineering platform for packaged desktop, mobile, web, native, managed, and console-style targets. It recovers shipped metadata, restores source from maps and resources, correlates external tool output, and prepares real project workspaces instead of stopping at raw dumps.

Why it stands out

Built for recovery, correlation, and execution, not just static dumping.

Source Recovery First

Recovers original source trees from shipped source maps, Tauri assets, BAML/XAML, package resources, and managed/native metadata where possible.

Real Format Breadth

Handles PE, ELF, Mach-O, WASM, APK, IPA, AppImage, CAB, MSI, JAR, AAR, DEX, .NET bundles, Electron, Tauri, and more.

External Tool Correlation

Ingests and correlates exports from Ghidra, rizin, radare2, ILSpy, JADX, and local symbol tooling into one queryable graph.

Operator Interfaces

Use the PyQt GUI for browsing, the CLI for automation, or MCP to let any connected LLM reason over artifacts and reconstruction workflows.

Runtime and Symbol Workflows

Supports remote PDB acquisition, runtime tracing, Frida hooks, native tool exports, package inspection, and framework-specific heuristics.

Rebuild and Patch Paths

Generates Android Studio, Xcode, Node, Tauri, Electron, and CMake-oriented workspaces with signing plans, patch bundles, and bounded package actions.

Workflow

From target file to reconstructed project with one continuous pipeline.

01

Identify

Classify the target, unpack installers and archives, fingerprint frameworks, and discover platform-specific packaging layers.

02

Extract

Recover manifests, resources, sections, symbols, source maps, managed assets, web bundles, package payloads, and runtime evidence.

03

Correlate

Normalize functions, strings, artifacts, frameworks, and tool exports into a unified analysis graph for browsing and automation.

04

Reconstruct

Restore original files when possible and optionally invoke GPT-assisted approximation only when the evidence demands it.

05

Execute

Move into rebuild templates, package actions, signing flows, patch bundles, and run-to-run diffs instead of ending with a static report.

Interfaces

Choose the control surface that fits the job.

GUI

PyQt5 desktop interface for browsing reports, artifacts, recovered sources, index entities, workflows, and runtime output.

CLI

Scriptable analysis, tool installation, diff generation, patch bundle creation, package actions, and background-friendly runs.

MCP

Expose the whole RE-Pro workflow to external LLM clients so they can inspect evidence, reconstruct files, validate changes, and drive rebuild steps.

re-pro analyze path\to\target.exe -o analysis_output
re-pro compare-runs path\to\base_run path\to\head_run -o diff_output
re-pro create-patch-bundle path\to\base_run path\to\head_run -o patch_bundle
re-pro mcp-server --transport stdio

Roadmap

The platform is already broad, but it is clearly being built toward something larger.

RE-Pro is moving beyond analysis into full reconstruction, rebuild, patching, and orchestration. The next major layers are richer package-aware rebuilders, deeper patch application, broader console and engine ecosystems, and more powerful graph-driven automation over the existing MCP and GUI surfaces.

In other words: this is not a one-off script collection. It is becoming a proper reverse-engineering platform.
